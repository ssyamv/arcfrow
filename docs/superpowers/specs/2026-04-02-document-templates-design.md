# 文档模板设计规格文档

> 版本：v1.0 · 2026-04-02

---

## 一、设计背景

### 问题

当前团队 PM 各自写 PRD，没有统一格式，也没有正式的 Review 流程。AI 生成技术设计文档时，输入质量不稳定，容易理解偏差。

### 目标

1. 定义统一的 PRD 模板，确保 AI 能准确理解需求
2. 定义技术设计文档模板，规范 AI 的输出结构，方便研发 Review
3. 降低 AI 理解跑偏的风险

### 设计决策

采用**分级模板**方案：按需求粒度提供两套 PRD 模板（模块级 / 功能点级），共享"AI 必读字段"确保理解准确性的下限。

---

## 二、PRD 模板

### 2.1 Frontmatter 规范（两套模板共享）

```yaml
---
title: 功能名称
type: module | feature
status: draft | active | deprecated
owner: 产品-姓名
created: 2026-04-01
sprint: S12
related_prd: []
---
```

| 字段 | 必填 | 说明 |
|------|------|------|
| title | 是 | 功能名称 |
| type | 是 | `module`（模块级）或 `feature`（功能点级） |
| status | 是 | `draft` 草稿 / `active` 生效 / `deprecated` 废弃 |
| owner | 是 | 负责 PM |
| created | 是 | 创建日期 |
| sprint | 否 | 所属迭代 |
| related_prd | 否 | 关联的其他 PRD 文件路径 |

### 2.2 AI 必读字段（两套模板共享）

无论模块级还是功能点级，以下三个字段是 AI 生成技术文档的最低输入：

| 字段 | 作用 | 填写要求 |
|------|------|----------|
| 一句话描述 | AI 理解需求方向的锚点 | 用"让[谁]能够[做什么]以达到[什么目的]"的句式 |
| 核心业务规则 | AI 生成逻辑判断的依据 | 用编号列表，每条规则必须是可判定的（有明确的是/否边界） |
| 验收标准 | AI 生成测试用例的依据，研发 Review 的对照基准 | 用"Given-When-Then"或"操作 → 预期结果"格式 |

### 2.3 功能点级模板（feature）

适用于单个功能迭代，如"添加手机验证码登录"。

```markdown
---
title: 
type: feature
status: draft
owner: 
created: 
sprint: 
related_prd: []
---

## 一句话描述

让[谁]能够[做什么]以达到[什么目的]。

## 背景

为什么要做这个功能？解决什么问题？（2-3 句话即可）

## 核心业务规则

1. [规则一：可判定的条件和结果]
2. [规则二]

## 功能说明

描述用户操作流程，包括：
- 入口在哪
- 主流程步骤
- 异常情况怎么处理（如输入错误、网络异常）

## 设计稿

- Figma 链接：[待设计师填写]
- 状态：未开始 | 设计中 | 已交付

## 涉及端

- [ ] 后端
- [ ] Vue3 Web
- [ ] Flutter 移动端
- [ ] Android 客户端

## 验收标准

1. Given [前置条件]，When [操作]，Then [预期结果]
2. ...

## 补充说明（可选）

UI 原型链接、参考竞品、与现有功能的关系等。
```

### 2.4 模块级模板（module）

适用于完整功能模块，如"用户注册登录系统"。在功能点级基础上增加模块级特有字段。

```markdown
---
title: 
type: module
status: draft
owner: 
created: 
sprint: 
related_prd: []
---

## 一句话描述

让[谁]能够[做什么]以达到[什么目的]。

## 背景

### 现状问题
当前存在什么问题？影响了谁？（3-5 句话）

### 目标
本模块要达成的核心目标，按优先级排列：
1. [目标一]
2. [目标二]

## 核心业务规则

1. [规则一]
2. [规则二]

## 功能清单

将模块拆解为具体功能点，标注优先级：

| 功能点 | 优先级 | 说明 |
|--------|--------|------|
| 功能 A | P0 必须 | 简要描述 |
| 功能 B | P1 应该 | 简要描述 |
| 功能 C | P2 可选 | 简要描述 |

## 用户角色与权限

| 角色 | 可执行操作 |
|------|-----------|
| 普通用户 | ... |
| 管理员 | ... |

## 各功能点说明

### 功能 A

**操作流程：**
- 入口在哪
- 主流程步骤
- 异常情况处理

### 功能 B
...

## 设计稿

- Figma 链接：[待设计师填写]
- 状态：未开始 | 设计中 | 已交付

## 涉及端

- [ ] 后端
- [ ] Vue3 Web
- [ ] Flutter 移动端
- [ ] Android 客户端

## 验收标准

1. Given [前置条件]，When [操作]，Then [预期结果]
2. ...

## 非功能需求（可选）

性能要求、并发量预估、数据量级、安全合规等，有明确要求才写。

## 补充说明（可选）

UI 原型链接、参考竞品、与现有模块的关系、数据迁移需求等。
```

### 2.5 两套模板的差异对比

| 字段 | 功能点级 | 模块级 |
|------|---------|--------|
| 一句话描述 | ✅ | ✅ |
| 背景 | 简要（2-3 句话） | 含现状问题 + 目标优先级 |
| 核心业务规则 | ✅ | ✅ |
| 功能清单 | ❌ | ✅（含优先级） |
| 用户角色与权限 | ❌ | ✅ |
| 功能说明 | 单个功能描述 | 按功能点拆分描述 |
| 设计稿 | ✅ | ✅ |
| 涉及端 | ✅ | ✅ |
| 验收标准 | ✅ | ✅ |
| 非功能需求 | ❌ | ✅（可选） |

## 三、技术设计文档模板

AI（Claude Opus）根据 PRD 自动生成，研发 Review 的对象。

```markdown
---
title: 
source_prd: /prd/2026-04/feature-xxx.md
status: draft | reviewed | approved
generated_by: claude-opus-4-6
generated_at: 2026-04-01T10:00:00
reviewer: 
---

## 功能概述

一段话总结：做什么、为谁做、核心价值。
（AI 基于 PRD 的"一句话描述"和"背景"生成，研发用来快速判断 AI 是否理解对了方向）

## 需求理解确认

AI 对 PRD 核心业务规则的理解复述：
1. [AI 理解的规则一]
2. [AI 理解的规则二]

> ⚠️ 如有以下疑问请 PM 确认：
> - [PRD 中未明确的边界情况]
> - [可能存在歧义的描述]

（防止 AI 理解跑偏的关键防线，研发 Review 时优先看这里）

## 数据库设计

### 新增/修改表

（建表 SQL，含字段注释、索引、关联关系）

### ER 关系说明

（表间关系描述）

## 接口设计

### 接口清单

| 接口 | Method | Path | 说明 |
|------|--------|------|------|
| 接口一 | POST | /api/v1/xxx | ... |

### 接口详情

#### 接口一：xxx

- 请求参数
- 响应结构（统一 Result<T>）
- 错误码
- 业务逻辑要点

## 分层实现说明

按 Controller → Service → ServiceImpl → Mapper → Entity 分层，说明各层职责和关键逻辑。

## 涉及的现有模块改动

列出需要修改的现有代码模块及改动点。（无则写"无"）

## 注意事项

- 安全性考虑
- 性能考虑
- 边界情况处理
- 与现有功能的兼容性
```

### 3.1 模板关键设计点

| 设计点 | 说明 |
|--------|------|
| 需求理解确认 | AI 先复述理解再给方案，研发 Review 第一步就能判断方向对不对 |
| 疑问列表 | AI 主动暴露不确定的地方，而不是猜测后默默给出可能错误的方案 |
| source_prd 字段 | 关联原始 PRD，方便回溯 |
| 仅覆盖后端 | 前端/客户端的实现指引在"多端代码生成策略"中单独设计 |

## 四、Figma 设计稿在流程中的位置

### 4.1 数据流调整

原流程：
```
PRD → 技术文档 → OpenAPI → 代码生成
```

新增设计稿环节后：
```
PRD → 技术文档 + OpenAPI（AI 生成，可与设计稿并行）
    → 设计师出 Figma 设计稿
    → AI 读 Figma + 技术文档 → 生成 UI 代码（前端/客户端）
    → AI 读 OpenAPI + 技术文档 → 生成后端代码
    → 研发 Review / 微调 → CI/CD
```

### 4.2 触发时机

- 技术文档生成不依赖设计稿，PRD Approved 即可触发
- UI 代码生成依赖设计稿交付（Figma 链接填写 + 状态改为"已交付"）
- 后端代码生成不依赖设计稿，可与设计稿并行

### 4.3 对其他模块的影响

| 模块 | 影响 |
|------|------|
| 胶水服务 | 需要对接 Figma MCP，在 UI 代码生成时读取设计稿 |
| 多端代码生成策略 | 拆分为后端生成和 UI 生成两条并行线 |
| Claude Code | 需要配置 Figma MCP Server，headless 模式下读取设计稿 |

## 五、文件存放规范

| 文件类型 | 存放路径 | 命名规则 |
|----------|----------|----------|
| PRD 模板 | `/prd/_template-feature.md` | — |
| PRD 模板 | `/prd/_template-module.md` | — |
| PRD 文档 | `/prd/{yyyy-MM}/{功能名}.md` | 按月份分目录 |
| 技术设计文档 | `/tech-design/{yyyy-MM}/{功能名}.md` | 与 PRD 同名 |
